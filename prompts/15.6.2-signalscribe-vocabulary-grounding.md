### Step 5 — Diagnose based on evidence

Now you know the shape of the problem. Two most likely patterns:

**Pattern A — Two-word phrases:** SignalScribe emits things like `["analytics portal", "reporting dashboard"]`. Registry has `analytics_portal`. The fuzzy matcher needs to normalize spaces ↔ underscores.

**Pattern B — Broken tokens:** SignalScribe emits `["analytics", "dashboard", "filtering"]` — individual words instead of compound identifiers. Even normalization won't save you — the concept is right but the expression is lossy.

**Pattern C — Completely different vocabulary:** SignalScribe emits domain descriptions like `["user interface", "export workflow"]`. The registry uses technical identifiers. Pure semantic mismatch.

Each has a different proper fix.

### Step 6 — Apply the right fix

**For Pattern A (space/underscore normalization):**

Update the pre-filter to normalize both sides before matching:
```python
def _normalize(s: str) -> str:
    return s.lower().strip().replace(" ", "_").replace("-", "_")
```

Apply to both `impact_areas` and `owned_product_areas` before set intersection. Keep the fuzzy fallback from 15.6.1 for edge cases.

**For Pattern B or C — the proper fix is prompt grounding:**

The real architectural issue is that SignalScribe doesn't know what vocabulary to use. Right now it invents English descriptions of impact areas. It should instead choose from a known vocabulary.

Fix: modify `.claude/agents/signalscribe.md` to inject the registry's canonical product area list. Something like:

```markdown
## Impact areas vocabulary

When you populate `impact_areas`, choose from this canonical list of product 
surfaces owned by business units. These are the only strings that downstream 
matching will recognize:

- `analytics_portal`
- `reporting_dashboard`
- `hcp_portal_ordering`
- `prior_authorization`
- `specialty_pharmacy_workflow`
- `medical_information_portal`
- `prescriber_attestation`
- `formulary_lookup`
- `exports`
- `clinical_operations_dashboard`
- ... (enumerate the actual full set from bu_registry.yaml)

If the change affects something not in this list, use the closest match and 
add a free-text note in `why_communicate` describing the actual surface. Do 
NOT invent new `impact_areas` values — unmatched values cannot be routed to BUs.
```

Read `config/bu_registry.yaml` and enumerate the actual vocabulary (union of all `owned_product_areas` across all BUs). Inject the resulting list into the SignalScribe prompt.

This is architecturally the right fix. Agents producing vocabulary downstream code consumes should be told what vocabulary to use. Pattern A's normalization is a workaround; Pattern B/C requires actually solving the contract problem.

**My strong preference: implement prompt grounding (Pattern B/C fix) regardless of which pattern the data shows.** Here's why:

- If it's Pattern A, grounding also fixes it (SignalScribe will emit exact identifiers, no normalization needed)
- If it's Pattern B or C, only grounding fixes it
- Grounding is more robust long-term — normalization will still fail for unexpected variants
- Grounding makes the agent-vs-code contract explicit, which is architecturally cleaner

**If you choose Pattern A's normalization fix:** keep grounding on the roadmap as v0.2 work.

**If you choose prompt grounding:** also keep 15.6.1's fuzzy match as a defensive fallback; don't rip it out.
