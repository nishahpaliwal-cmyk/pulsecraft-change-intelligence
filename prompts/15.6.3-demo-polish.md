## Issue 1 — ARCHIVE path leaves empty skeleton placeholders

**Symptom:** When SignalScribe hits ARCHIVE at Gate 1, only the ARCHIVE decision card should render. Currently, two empty gray skeleton boxes render above it where Gates 2 and 3 would have appeared if the pipeline had continued. They never resolve because those gates never fire. The UI looks broken — like something is still loading.

**Visual evidence:** Scenario 02 (pure internal refactor) shows two large gray rectangles above the ARCHIVE card.

**Fix:**

The SignalScribe section renders three gate placeholders optimistically when the section mounts, expecting all three to fill. When only one gate fires (because of early ARCHIVE), the other placeholders should be removed, not left hanging.

Approach: render gate placeholders only when a `gate_decision` event for that gate arrives. Don't pre-render all three. Or, if pre-rendered is easier structurally, detect the ARCHIVE decision and remove unfired placeholders when the terminal state arrives.

In the event stream, when a `signalscribe_decision` event with `verb: "ARCHIVE"` arrives, or when `terminal_state: ARCHIVED` is received before later gates fire, remove any unfilled gate placeholders from the DOM.

Verify: run scenario 02. Only one gate card should appear in the SignalScribe section — the ARCHIVE card. No gray boxes above or below.

The larger heading draws the eye and the smaller one reads as a bug.

Option B: keep the giant heading as the state title, remove the section divider entirely for terminal states.

Implement Option A unless the code structure strongly prefers B. Either way: **"Awaiting Human Review" must only appear once** in the rendered document.

Verify: run scenario 01, 03, and 04. Each should show exactly one "Awaiting" or equivalent heading in the terminal section.

## Issue 3 — PushPilot section missing entirely

**Symptom:** The architecture places PushPilot between BUAtlas and pre_deliver — it owns Gate 6 (schedule and channel selection). But the current UI shows the pipeline going directly from BUAtlas to the HITL routing banner (or to DELIVERED/HELD), with no PushPilot section anywhere.

**Visual evidence:** No screenshot shows a PushPilot section at all. After BUAtlas cards, the next thing rendered is the pre_deliver "Routed to human review" banner.

**Fix:**

Two sub-issues to investigate:

**A) Is PushPilot actually running?**

Check the orchestrator flow. Does `engine.run_change` invoke PushPilot after BUAtlas returns? Does `instrumented_run.py` emit events for it?

Run a scenario and search the audit log:
```
grep -l pushpilot audit/$(date +%Y-%m-%d)/*.jsonl | head -3
```

If PushPilot audit records exist, it's running but events aren't being emitted to the frontend. Fix: emit `pushpilot_invoke_start`, `pushpilot_decision`, and `pushpilot_invoke_complete` events in `instrumented_run.py`.

If PushPilot audit records don't exist, it's not running. That's a bigger concern. Look at `engine.run_change` — is the PushPilot call commented out, conditional on something unusual, or short-circuited for HITL paths?

**B) Is the PushPilot UI section coded but not receiving events?**

Search `app.js` for PushPilot handling:
```
grep -n -i "pushpilot\|push_pilot\|push-pilot" src/pulsecraft/demo/static/app.js
```

If there's no PushPilot rendering code, add it. If there is, the events aren't reaching it.

**Build the PushPilot section rendering:**

Similar styling to SignalScribe but with coral accent (per spec):
- Section divider: "PushPilot" in Fraunces 500 18px, subtitle "Gate 6" in Inter 400 13px tertiary color, coral left bar (3px)
- Decision card (same structure as SignalScribe decision cards): verb badge on left, "Gate 6 · timing" label, confidence bar on right, reasoning paragraph below

**For the agent-vs-code divergence case:** when PushPilot's preferred decision differs from pre_deliver's enforced decision, render both cards stacked with the arrow connector as specified in 15.6. If they match, render a single decision card (most common case).

A divergence happens when:
- PushPilot says SEND_NOW but pre_deliver says HOLD_UNTIL (quiet hours)
- PushPilot says SEND_NOW but pre_deliver says HOLD_UNTIL (mlr_sensitive)
- PushPilot says DIGEST but pre_deliver says SEND_NOW (priority override)

Detection logic: compare `pushpilot_decision.verb` with the terminal routing trigger. If they differ, render divergence layout. Otherwise single card.

**Verify with scenario 04:** this fixture should produce divergence — PushPilot likely says SEND_NOW or DIGEST, pre_deliver says HOLD for mlr_sensitive reason. If divergence renders, the architectural showpiece lands.
