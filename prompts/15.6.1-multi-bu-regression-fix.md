## Outcome (session 15.6.1 — 2026-04-24)

**Root cause:** LLM vocabulary nondeterminism, not a code regression. SignalScribe produced plain-language `impact_areas` like `["analytics", "reporting", "dashboard"]` in the 23:37 real-agent run instead of exact registry identifiers like `["analytics_portal", "reporting_dashboard"]`. The pre-filter uses exact string intersection, so zero candidates were found.

**Files changed:**
- `src/pulsecraft/skills/registry.py` — added `_norm()` helper and keyword-match fallback pass to `lookup_bu_candidates`; the fallback runs ONLY when the exact pass finds nothing, normalises both BU keywords and impact_areas to lowercase+space, and checks equality
- `tests/unit/skills/test_registry.py` — updated `test_keywords_not_used_in_matching` → `test_keywords_used_as_fallback_when_no_exact_match`; added `test_keywords_fallback_not_used_when_exact_match_exists`
- `tests/integration/orchestrator/test_run_change_fixtures.py` — added `test_analytics_vocabulary_does_not_produce_no_candidate_bus` regression guard

**Regression localization:** No specific commit introduced a code regression. The 15.6 commits were purely frontend (confirmed by `git diff 67f58eb..HEAD -- config/ src/pulsecraft/`). The failure was LLM nondeterminism surfacing in a real-agent run.

**Scenario results (with mock agents — real-agent runs require API key):**
- 001: 2 BUs evaluated (bu_alpha, bu_epsilon), terminal state HELD (quiet hours; AWAITING_HITL in real-hours runs)
- 006: 2 BUs evaluated (bu_alpha, bu_epsilon via mock impact_areas), terminal state HELD — regression guard test verifies analytics vocab finds ≥1 BU
- 007: 2 BUs evaluated, terminal state HELD
